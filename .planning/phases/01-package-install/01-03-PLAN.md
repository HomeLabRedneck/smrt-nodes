## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| developer→game process | Launching game subprocess; cannot directly inspect runtime memory |
| game→modloader.log | Log is the only programmatic signal of mod state; trust depends on log being fresh |
| user→checkpoint response | Human attestation is the only way to verify in-game UI state |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-11 | Spoofing | Stale modloader.log from previous session could falsely pass verification | mitigate | Task 1 captures pre-launch mtime and confirms post-launch mtime is newer |
| T-01-12 | Tampering | False positive in log regex (e.g., a SUCCESS line for a different mod that contains "kuuk") | mitigate | Regex is narrow: `SUCCESS\s+kuuk:STM:Main:\s+Initialized` matches only the exact prefix used by these mods |
| T-01-13 | Information Disclosure | modloader.log may contain user paths (already in scope: %APPDATA%, game install) | accept | These paths are already known to the user and are local; no remote disclosure |
| T-01-14 | Denial of Service | Game crashes on launch before writing log | mitigate | Acceptance criteria require log to exist AND have post-launch mtime; absence triggers visible failure |
| T-01-15 | Repudiation | User claims windows worked but actually didn't (or vice versa) | accept | This is a solo developer workflow; user's own attestation is the ground truth |
</threat_model>
